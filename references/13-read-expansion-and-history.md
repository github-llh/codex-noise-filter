# 读取扩窗与历史防回归

本文件按需读取。只有任务涉及读取行数限制、智能扩窗、完整语义单元、强规则漏判、Git 历史、提交记录、回归风险或旧逻辑兼容时打开。通用执行门禁、任务胶囊、调用链和局部对齐仍见 `02-noise-filter-workflow.md`。

## 读取完整性与智能扩窗

读取窗口是 token 预算工具，不是规则判断边界。只要准备修改代码、判断是否同步治理旧代码、或评估强规则是否低风险闭环，就必须先按当前文件类型、语法结构、框架配置和对应技术栈 reference 自动确认当前已读内容是否覆盖了本次语义单元。这个规则适用于 Java、Python、Vue/React、小程序、SQL、配置、脚本和测试，不等待用户指定触发。

必须扩窗的触发信号：

- 已读窗口内出现方法、类、函数、组件、页面、hook/composable、SQL、配置块、脚本入口、测试 fixture 或模板片段被截断，无法看到完整开始/结束边界。
- 已读窗口内出现硬编码、重复 if/set、重复映射、配置写死、分层错位、缺注释、安全边界缺口等强规则信号，但相关声明、常量、helper、调用方或被调方不在窗口内。
- 修改点引用了未读字段、常量、枚举、DTO/VO、Python 类型/docstring、props/emits/slots、hook/composable、store、route、配置 key、SQL 片段、测试 fixture、页面配置、template/wxml/wxss 或工具函数。
- 同一文件已有明显范式，例如相邻方法、同类函数、同类字段、同类组件、同类页面、同类测试、同类枚举、同类 SQL 或同类配置块，但当前窗口没有覆盖对照样例。
- 只读了 diff 附近几行，却需要判断“是否已有相同逻辑”“是否应抽常量/枚举”“是否 Controller 写了业务逻辑”“是否缺注释/校验/事务边界”。
- 用户明确指出“没读到”“读取行数限制”“漏判”“无法自动更改”等读取完整性问题。

扩读顺序：

1. 自动识别技术栈和语义单元：按扩展名、配置文件、框架依赖、注解/装饰器、组件语法、路由/页面配置、测试框架和 mapper/config 结构判断，不把 Java 方法当成唯一模型。
2. 定位符号或结构边界：优先用 IDE 符号、语言服务器、结构搜索、AST/语法工具；不可用时用 `rg`、`ctags`、括号/缩进/标签/JSON/YAML/TOML/XML 结构和框架配置确认起止行。
3. 读取本次修改语义单元的完整逻辑；如果完整体超过常规窗口，分段读取同一单元，不用一次性读全文件。
4. 按技术栈读取关键声明和契约区：
   - Java：imports、注解、字段、常量、枚举、DTO/VO/Entity、Service 接口、事务和 mapper/SQL。
   - Python：module imports、常量、dataclass/Enum、函数/类/docstring、typing、配置对象、pytest fixture。
   - Vue/React：props/emits/slots、state、hooks/composables、store/router、template/JSX、样式作用域和 api client。
   - 小程序：`Page`/`Component`、`data`、`properties`、`methods`、`setData`、`app.json`/`pages.json`、wxml/wxss/wxs、分包配置。
   - SQL/配置/脚本/测试：完整 mapper statement、动态 SQL、配置块、脚本入口、参数解析、fixture 和断言边界。
5. 读取直接调用点和被调用 helper 的最小窗口；若调用链已在其他文件，按调用链规则继续最小扩读。
6. 读取同文件相邻 1 到 3 个同类范式，用来判断项目风格、命名、注释、Lombok/typing/props、校验、错误处理、状态管理和抽象方式。
7. 若强规则判断仍不闭环，再用 `rg` 精确搜索同名常量、魔法值、重复片段、方法名、配置 key、状态值、路由/页面名或测试名，只读取命中的最小窗口。

不能做的事：

- 不能因为默认窗口是 200 到 300 行，就在未读完整语义单元时断言“没有明显问题”。
- 不能等待用户指定“扩哪几行”才处理；除非语法边界、生成来源或业务语义无法自动判断，否则必须主动扩读能证明闭环的最小区间。
- 不能把智能扩窗写死成 Java 专属；每个技术栈都按自身语法规范、抽象要求和对应 reference 判断。
- 不能为了扩窗直接全仓扫描或全文件通读大型文件；先按符号、调用链、相邻范式和精确搜索逐步扩读。
- 不能把同文件所有历史问题都纳入本轮；只处理本次修改单元、直接调用链和为完成任务必须读取的相关区域内的低风险闭环问题。

扩读后的决策：

- 若扩读后确认强规则违背位于触碰范围、直接调用链或必须读取的相关区域，按 `02-noise-filter-workflow.md#强规则命中后的自动升级` 写入任务胶囊并评估同步处理。
- 若扩读后确认问题在无关历史区域，记录为风险或后续项，不扩成全模块重构。
- 若文件过大、语法不完整、生成文件、二进制/压缩产物、框架生成链路不清或工具无法定位语义边界，说明未读边界和保守处理策略；不得把未验证区域当成已检查。只有这种无法自动判断的情况才需要用户补充。
- 最终回复只汇报关键扩读结论，不搬运大段代码。

## Git 历史对比与回归防护

Git 历史是判断既有行为语义和回归风险的证据，不是每次任务都全量翻仓库。只要修改会影响旧逻辑、公共契约、兼容行为或高风险边界，就必须自动读取最小历史证据，避免因为只看当前任务需求而把历史约束改崩。

必须自动查历史的触发信号：

- 修改涉及 API、DTO/VO/Entity、数据库字段、权限、路由、配置键、序列化值、状态码、枚举值、外部协议、消息格式、缓存 key、任务调度、事务边界或安全边界。
- 准备删除、替换、抽象、合并旧逻辑，或把旧逻辑迁移到枚举、策略、配置、helper、hook/composable、service、mapper、SQL、脚本或测试中。
- 当前需求和现有代码冲突，或现有代码看起来“多余/奇怪/绕”，但可能是兼容历史 bug、第三方约束、灰度开关、数据迁移、老版本客户端或平台限制。
- 文件或同一语义单元最近多次改动、反复修 bug、回滚过、出现 revert/fix/hotfix/兼容/临时/历史等提交信息。
- 构建、测试或线上问题指向“最近一次变化点”，需要判断本次是否应恢复、兼容或重新实现旧行为。
- 用户提到“改崩”“回归”“之前可以”“看提交记录”“为什么这样写”“不要只看当前需求”等风险信号。

历史读取顺序：

1. 先确认当前工作区状态：`git status --short -uall`，区分用户未提交改动、本轮改动和未跟踪文件；不得覆盖用户改动。
2. 对目标文件读取最小提交列表：`git log --oneline --decorate -- <file>`；重命名风险明显时加 `--follow`。
3. 对本次语义单元读取行级历史：`git blame -L <start>,<end> -- <file>` 或 IDE annotate；只看触碰行和相邻关键声明。
4. 对关键状态值、方法名、配置 key、SQL、协议字段、错误码、路由、枚举值等用 `git log -S'<token>' -- <path>` 或 `git log -G'<pattern>' -- <path>` 查引入、修改和删除原因。
5. 只打开与本次风险相关的少量提交：`git show --stat --summary <commit>` 先看范围，再用 `git show <commit> -- <file>` 看具体 diff；必要时看同提交相关文件以理解契约变化。
6. 若当前文件是新文件、未跟踪、生成文件或历史不足，转而查调用方、被调方、配置入口、测试或旧文件名的历史。

历史证据判断：

- 若历史显示旧逻辑是兼容、灰度、迁移、权限、安全、第三方协议、老数据或平台限制，修改必须保留兼容路径或明确迁移边界，不能直接删除。
- 若历史显示旧逻辑是临时绕过、重复补丁、已废弃兼容或历史 bug 修补，可在低风险闭环内清理，但要保持外部契约和验证路径。
- 若历史提交只说明格式化、重命名、机械迁移，不足以证明语义，可继续看相邻提交、测试或调用链，不把格式化提交当业务意图。
- 若历史证据和当前需求冲突，先选择兼容保守方案；需要改公共契约、数据库、权限、安全或外部协议时，说明风险并等待用户确认。

不能做的事：

- 不能为了“了解历史”全量读所有提交、所有文件或长期时间线；只围绕触碰文件、语义单元、关键 token 和直接调用链查最小历史。
- 不能用 `git checkout`、`git reset`、`git restore` 等会改工作区的命令读取历史；读取历史用 `git show`、`git log`、`git blame`。
- 不能把历史提交作者当责任判断；只使用提交信息和 diff 解释代码意图。
- 不能因为用户没有要求查历史，就跳过明显回归风险场景。

输出与任务胶囊：

- 任务胶囊记录是否查历史、触发原因、关键提交或 blame 区间、历史意图和仍未闭环的风险。
- 修改前用一句话说明历史对比结论，例如“该分支逻辑由兼容旧客户端引入，本次保留入参兼容，只迁移内部重复映射”。
- 最终回复只汇报与本次修改有关的历史结论，不铺陈完整提交日志。
