# 全局工程规则

本文件只保留所有编程任务都需要先看的通用规则。各技术栈 reference 只保留落地差异；文件归属、环境命令、验证、安全边界、编码与中文乱码、硬编码、重复逻辑和注释先按本文件判断，再进入对应技术栈文件。

## 语言偏好

- 默认使用简体中文回复。
- 分析、解释、计划、变更说明、验证结果、根因分析、下一步建议都用简体中文。
- 代码、命令、配置项、环境变量、文件名、类名、方法名、日志、异常信息保持原文。
- 不使用英文标题 `What Changed`、`Validation`、`Root Cause`、`Next Steps`、`Summary`。
- 使用中文标题：`变更内容`、`验证结果`、`根因分析`、`下一步`、`总结`。

## 跨技术栈编码与中文乱码门禁

这是内部自动门禁。只要新增、修改、读取、检索、格式化、构建、测试、lint/typecheck、终端输出、页面渲染、日志、配置或第三方转发载荷中出现中文字符、非 ASCII 文案、乱码、替换字符、`encoding`、`charset`、`UTF-8`、`GBK`、locale、跨平台文件名或字符集相关错误，就必须先确认编码来源和验证路径，不能把它当成普通展示问题跳过。

触发信号：

- 文件内容或 diff 中出现 `�`、`???` 替换、中文被拆成乱码、转义异常、字符串截断、非 ASCII 文件名异常或日志里中文不可读。
- 编译、构建、测试、lint、typecheck、打包、运行输出包含 `encoding`、`charset`、`MalformedInputException`、`UnmappableCharacterException`、`UnicodeDecodeError`、`UnicodeEncodeError`、`iconv`、`file.encoding`、`project.build.sourceEncoding`、`meta charset`、`Content-Type` charset 等信号。
- Maven/Java、Node/前端、Python、小程序、终端 Shell、CI、IDE、第三方 agent 或模型路由环境可能使用了不同 locale、默认编码、换行或文件名清洗规则。

处理顺序：

1. 先确认项目事实，不凭个人编辑器默认值判断：`.editorconfig`、`pom.xml` 的 `project.build.sourceEncoding`、Maven compiler/resources 配置、前端 `meta charset`/响应头、TypeScript/构建配置、Python 源文件声明、README/CI 脚本和已有同类文件编码。
2. 读取当前文件时保留原编码证据；不要用整文件重写、格式化或复制粘贴无意改变换行、BOM、缩进和非 ASCII 字符。
3. 若进入工具链命令节点，先按 `06` + `14` 解析 active cache path，并记录会影响编码的工具版本、脚本、locale、source encoding、前端规范文件或 Python 解释器。
4. 修复乱码优先在正确边界处理：源文件编码声明、构建资源编码、终端/CI locale、HTTP `Content-Type` charset、HTML meta、i18n 资源、后端响应编码、文件名 slug 规则或数据源解码；不要在业务逻辑里硬编码二次转码掩盖根因。
5. 如果第三方工具声称已修复乱码，仍要用当前文件、当前 diff 和最小命令确认没有引入双重编码、转义丢失、展示文案变更或跨平台文件名风险。

落地边界：

- Java/Maven：优先检查 `project.build.sourceEncoding`、`maven.compiler.encoding`、resources filtering、运行时 `file.encoding` 和日志编码；不要把中文常量或注释在不同编码间来回转换。
- 前端/小程序：优先检查 `.editorconfig`、Prettier/ESLint、HTML/页面 `charset`、HTTP 响应头、i18n 资源、构建工具和小程序平台配置；不要把中文展示文案当业务 code 判断值。
- Python：优先检查源文件编码声明、解释器版本、终端 locale、文件读写 `encoding=` 和测试 fixture 编码；文件 IO 必须显式使用项目需要的编码。
- 跨平台文件名：缓存 profile、生成文件、导出文件和临时文件必须使用 `06#跨系统缓存文件命名` 的清洗规则；中文名可作为业务展示值，但不能作为跨系统唯一标识。

验证要求：

- 修改源码或配置后，至少用最轻量方式确认当前文件可正确读取，相关构建/编译/lint/typecheck 或页面/日志输出不再乱码。
- 无法运行命令时，说明缺少的环境、locale、工具链、服务或样例数据，并保留已确认的文件编码证据。
- 不能只说“已改为 UTF-8”；必须说明依据来自项目配置、文件内容、命令输出或 active cache path。

## 工具优先级

JetBrains 项目必须优先使用 JetBrains MCP / IDE 工具完成代码读取、定位、修改和诊断。

- 只有 JetBrains MCP 明确不可用、超时或返回错误时，才允许使用 Shell。
- 禁止未尝试 JetBrains MCP 就直接用 Shell 扫描或修改 JetBrains 项目。
- 优先分析最小范围，不做无意义全仓扫描。
- 默认不使用 Sub Agent；只有当前任务目标需要并行调查、跨域隔离或已存在明确授权时才使用。
- 任意第三方调用、第三方 agent、App、终端/TUI/CLI、IDE 插件、MCP/ACP、hook、subagent、CI/chatops/webhook、`cc switch`、model/provider router、gateway/proxy/adapter、自定义 wrapper、未知转发层或模型路由只是入口或转发层，不改变工具优先级。JetBrains 项目仍先尝试 JetBrains MCP / IDE 工具；非 JetBrains 项目仍按本地文件工具、项目脚本和最小验证处理。
- 其他 agent 或路由器输出只作为证据线索；涉及读取、修改、格式化、lint/typecheck/build/test 或验证结论时，必须回到当前工作区真实文件、当前 diff、项目配置和 active cache path 重新确认。

非 JetBrains 项目可直接使用本地文件工具。搜索优先 `rg` 或 `rg --files`。

## 修改前确认

修改前必须确认：

1. 目标文件。
2. 问题根因。
3. 最小修改方案。
4. 不应受影响的模块、接口、数据结构、权限和业务逻辑。

修改前再次引用任务编号确认目标。若任务描述与实现方向不一致，立即停止并汇报。

## 核心规则

- 优先最小、安全、可回滚的修改。
- 默认保持现有行为不变；只有当前任务目标、根因证据和影响评估共同指向需要改变行为时，才改变行为。
- 不修改无关文件，不随意重构正常工作的代码。
- 最小改动不是“只改新增行”或“只记风险”的豁免理由；当本次触碰范围、直接调用链或为完成任务必须读取的相关文件已经命中硬编码、重复逻辑、配置写死、分层错位、注释缺失、安全边界或验证缺口等明确治理项时，先判断调用链深度、涉及文件数量、契约风险和验证路径。影响面浅、文件少、兼容性明确时，必须写入当前任务清单和任务胶囊并直接做局部对齐；风险高或语义不闭环时才保留兼容并说明后续项。
- 不随意修改 API、DTO、数据库字段、权限、路由、配置键和公共契约。
- 未明确批准不要新增依赖。
- 未批准不要执行数据迁移、删除操作、网络写入和影响工作区外部数据的命令。
- 参考类似代码风格以及文件存放位置；新建文件归属地规则见 `07-java-backend-architecture.md#新建文件归属地`。

## 通用质量

- 优先复用现有模式，避免硬编码，保持接口清晰，保留未来扩展点。
- 做好输入校验，保留认证、权限、租户隔离和审计逻辑。
- 处理空值、异常值和边界情况，不吞异常。
- 避免 N+1 查询、全表扫描和无分页查询。
- 保持模块职责单一，优先清晰命名。
- 复杂逻辑说明原因而不是描述代码行为。

## 跨技术栈编码风格智能化门禁

写代码前先做风格预检，不能等实现写完后才被动清理魔法值、重复分支和常量散落。只要新增或修改代码、配置、模板、脚本、测试，就先判断本次会写入或继续保留的字面量、分支和重复结构应该放在哪里。

预检顺序：

1. 识别即将写入的字符串、数字、布尔开关、日期/时间格式、状态值、类型值、来源值、平台值、协议字段、错误码、配置 key、header、content type、路由名、事件名、storage key、阈值、时间窗和默认值。
2. 判断值的变化方式：固定闭合集合、技术标准常量、环境或运维可变值、运行期业务可维护值，还是短方法内自解释的局部字面量。
3. 区分“协议字段名/key”和“业务取值/value”：JSON 字段名、模板参数名、header name、storage key、route name、event name 通常是协议常量或契约常量；状态、类型、来源、渠道、平台、动作、模式、结果这类有限取值才优先枚举/联合类型/字典对象。不能把所有 `String` 常量机械改成枚举，也不能把业务取值伪装成普通常量。
4. 先查现有枚举、常量模块、配置属性、字典体系、i18n、design tokens、api client、SDK 常量、OpenAPI/生成类型或协议定义；命中已有模式必须复用，不重复造平行体系。
5. DTO/VO/Request/Response、OpenAPI/Swagger 注解、JSDoc/docstring 或字段注释里已经列出有限 code/label 取值时，先按字段语义、code 列表和说明检索现有枚举、联合类型、字典或生成类型；找到后必须复用它的 code、反查和校验能力，不再散写字符串或数字。
6. 根据业务域命名和归属放置：跨模块契约放契约层，业务内部值放领域或业务模块，页面私有值只在页面局部集中，测试 fixture key 跟随测试工具或 fixture builder。
7. 对多状态、多类型、多平台、多格式、多字段映射和重复默认值，先评估枚举行为、策略表、mapper/converter、schema、hook/composable、builder 或小型 helper，而不是继续复制 `if/switch/set`。
8. 确认兼容性：抽取常量、枚举或配置不能改变 API code、数据库值、序列化值、前端展示、路由、权限、外部协议或历史 mock；有风险时只先集中表达并说明边界。

禁止写法：

- 因为“只用一次”就把业务状态、外部协议、配置 key、路由名、事件名、错误码、阈值或超时直接散写到业务逻辑中。
- 看到 `private static final String` 或常量名就直接判定“满足规则”；必须继续判断它是协议 key、技术常量、配置值、固定业务取值，还是运行期可维护值。
- DTO/VO/API 字段说明已经明示固定取值，或项目已存在同业务语义的枚举/字典，却继续在赋值、比较、校验、mock 或默认值中写裸字符串/数字。
- 因为“最小改动”继续复制同一类 `if/switch/set/map`，导致新增一种状态或渠道时需要改多个位置。
- 新建 `CommonConstants`、`GlobalEnum`、`StatusEnum` 这类无业务归属的垃圾桶式常量/枚举。
- 把环境 URL、密钥、token、appid、超时、重试、线程池、缓存 TTL、灰度开关等运维可变值写成业务类常量。
- 在前端模板、JSX、wxml 或测试 mock 里用裸字符串承担业务判断。

允许边界：

- `0`、`1`、空字符串、布尔值、很短方法内的局部循环边界、测试里一次性且语义自解释的 fixture 值，可以保留为局部字面量。
- 协议字段名、模板参数名、header name、route name、event name、storage key 如果只是作为 map/json/config 的 key 使用，可保留为命名清晰的常量；同一协议 key 的固定取值如果参与业务判断或跨层传递，仍应枚举/字典化。
- 抽象会明显降低可读性、引入跨域依赖或改变公共契约时，不强行抽象，但要在任务胶囊或最终回复说明为什么保留。
- 两处以内且业务语义不同的相似代码可以暂不抽象；如果变化点稳定或预计会继续增加，必须先做抽象评估。

## 跨技术栈抽象抽离时机

这是内部自动门禁，不依赖用户说“抽公共方法”或“不要过度抽象”。新增、修改、阅读、检索、lint/typecheck 修复、截图/diff 审查、调用链确认或重构旧代码时，只要看到公共接口、公共方法、公共类、公共文件、helper、base 类、hook/composable、schema、mapper、converter、adapter、策略表、handler map、泛型、`any`、`Object`、反射或重复逻辑收敛意图，就必须先判断抽象时机，再决定是否写入抽象。

先满足以下条件，再抽离公共能力：

1. 复用点有稳定共同语义，不只是代码长得像；调用方使用同一抽象后，业务意图更清楚。
2. 变化点可预测，例如状态、类型、来源、平台、字段映射、校验、默认值、展示组装、请求适配或外部系统处理会继续增加。
3. 依赖方向正确，抽象不会让低层依赖高层、公共包依赖业务实现、API module 反向依赖实现 module，或让前端共享层依赖页面私有状态。
4. 公共契约能被类型、注释、测试或最小验证覆盖；抽象后不模糊权限、租户、审计、事务、异常、序列化、路由、缓存和外部协议边界。
5. 项目没有可复用的既有组件、类型、枚举、mapper、converter、schema、hook/composable、service、helper、SDK 类型或框架扩展点。

暂不抽离的情况：

- 只有一处使用，或两处以内但业务语义不同、变化方向不同。
- 抽象只减少几行代码，却引入跨层依赖、万能工具类、过宽类型、复杂泛型、反射、动态执行、隐式副作用或调试跳转成本。
- 为了配合抽象需要修改公共 API、DTO、数据库字段、权限、路由、序列化值、外部协议或历史兼容行为，而当前任务没有授权。
- 前端只是页面私有展示细节，小程序只是单页面局部 `setData` 路径，后端只是当前 Service 内一次性直观编排。

类型边界：

- 前端抽出的公共接口、组件、hook/composable、adapter、service 或工具方法，默认仍要优先使用明确类型、泛型、联合类型、组件库类型、生成类型、`unknown` + 收窄或运行期校验。`any` 只允许隔离在未类型化第三方库、框架透传层、迁移暂存层、无法表达的高度动态事件/slot/children/属性透传等最小边界，不能外泄到业务 props、请求/响应模型或公开 API；保留时必须在任务胶囊说明原因、作用域和后续收敛点。
- 后端抽出的公共接口、方法、类或文件，优先用清晰业务接口、枚举策略、显式 helper、编译期 mapper/converter 或受约束泛型保留输入输出类型关系；不要用裸 `Object`、原始类型集合、反射或复杂泛型逃避建模。泛型只在能提升编译期安全和复用稳定性时使用，不能掩盖事务、权限、租户、审计和异常边界。

## 跨技术栈文件归属与依赖边界

任何技术栈新增、移动或拆分文件前，都要先确认“归属地”，再动手写文件。不要因为当前目录打开方便、文件名相近或功能临时可用，就把代码放到 root、错误 module、构建产物目录或依赖方向错误的位置。

通用确认项：

1. 目标文件属于哪个业务域或技术域。
2. 它是源码、测试、配置、脚本、模板、迁移、资源，还是构建产物。
3. 它属于哪个 module/package/workspace/app/subpackage。
4. 同类文件当前放在哪里，命名和导入方式如何。
5. 新文件会被谁依赖，它又依赖谁，是否造成反向依赖、循环依赖或跨层访问。
6. 生成目录和源码目录是否已分清，例如 `dist/`、`build/`、`target/`、`unpackage/dist/`、`miniprogram_npm/`、`generated/`。

处理原则：

- 先查同业务域、同后缀、同职责文件；优先放到既有模式下，不新造并行目录。
- 接口/契约、实现、实体、配置、脚本、测试和样例可能分属不同 module；不能只按文件名相似放置。
- 测试跟随被测对象所在模块和项目已有测试目录；不要为了运行方便放到聚合 root 或生产包里。
- 构建产物默认只读不改；只有当前任务目标本身是分析产物、修复生成结果或无法从源码定位问题时才触碰产物，修复应优先回到源码和构建配置。
- 归属不确定时，先列出候选位置和依据；不能确认依赖方向时不直接新建。

## 跨技术栈环境与命令

运行、构建、测试和安装依赖前，先按项目声明选择工具链，不凭个人机器习惯写命令。

通用顺序：

1. 读取项目缓存、README、脚本、CI、IDE/项目配置、版本文件和 lockfile。
2. 使用项目声明的运行时、包管理器、构建工具和 workspace/filter 方式。
3. 本机路径未知时按 `06-environment-discovery.md` 发现、验证并缓存到当前工作区 active cache path，优先使用 `.codex/local-environment.<profile>.json`。
4. 缓存与项目配置冲突时，以项目配置为准并重新验证。

约束：

- 不混用包管理器、解释器、JDK、Maven、Node、Python、模拟器或 CLI。
- 不为了临时通过而全局安装依赖、修改用户 shell 配置、修改 IDE 全局配置或写死个人绝对路径。
- 构建命令必须在正确 root/workspace/module 运行；多模块和多包项目先确认聚合 root 与目标模块。
- 命令输出只保留与任务相关的结论、错误、路径和版本，不搬运大段日志。

## 跨技术栈验证策略

验证要和触碰范围、风险等级、项目已有命令匹配。不要假验证，也不要为了一个局部修改无差别跑昂贵全量流程。默认只做非交互命令验证，不做运行态、交互态或屏幕级操作验证。

优先级：

1. 语法、类型、导入、配置可解析等最快失败检查。
2. 该技术栈的最小编译或构建，例如 Maven `compile/package -DskipTests`、前端 `build/typecheck`、Python `compileall/py_compile`、小程序构建脚本。
3. 项目已有 `lint`、`typecheck` 或等价静态检查。
4. 触碰测试本身、当前任务目标是复现 bug、或编译/语法无法覆盖关键逻辑时，才补定向测试。
5. UI、浏览器、小程序模拟器、真机、外部系统、发布链路、接口手工调用、桌面点击和其他操作验证默认不执行；只有任务目标本身就是这类验证、验收必须依赖交互证据，且权限与副作用边界清楚时才执行。

要求：

- 优先使用项目已有 `compile`、`build`、`typecheck`、`lint`、CI、Makefile、tox/nox、workspace script 或框架命令；没有专门编译命令时，用该语言最小语法检查替代。
- Java、Python、前端、小程序和其他技术栈默认都不启动浏览器、不使用 Browser/Computer Use、不操控电脑屏幕点击、不打开模拟器或真机、不手工调用外部系统/API 页面。编译、语法、类型检查或构建通过即可作为默认验收；只有当前任务目标本身是运行态验证、页面点击、截图、视觉回归、E2E、模拟器、真机、外部服务联调或电脑屏幕操作，且权限与副作用边界清楚时，才执行对应操作验证。
- 无法运行验证时，说明缺少的环境、账号、密钥、模拟器、服务、数据或权限，不用静态推测冒充通过。
- 修改 bug 时默认先保证语法、编译、类型检查或构建通过；只有当前任务目标是测试修复、需要复现 bug、或编译/语法无法覆盖关键风险时，才补或运行能复现问题的定向测试。没有操作验证时说明默认边界，不用人工点击或外部联调冒充通过。
- 验证失败先区分本次改动、历史失败、环境缺失和命令选择错误，不连续盲改。

## 跨技术栈安全与外部边界

涉及认证、授权、租户、审计、密钥、外部 API、文件、网络、支付、消息、上传下载、富文本、动态执行或生产配置时，默认按高风险边界处理。

通用规则：

- 不把密钥、token、账号、生产 URL、上传凭证、私钥、机器人编号、白名单和真实 appid 写进代码、测试、README 或日志。
- 不绕过认证、授权、数据权限、租户隔离、审计、脱敏、限流和重复提交防护。
- 外部调用必须有超时、错误处理、重试边界、幂等或去重策略；不能无限重试或吞异常返回成功。
- 文件路径、上传文件、压缩包、模板渲染、SQL、命令参数、动态 URL、富文本、反序列化和动态执行要做输入边界检查。
- 不在事务内直接执行不可回滚副作用；需要提交后动作时使用事务事件、afterCommit、outbox、可靠消息或项目已有机制。
- 示例、测试和 mock 可以使用假值，但要明显不是可用凭证，且不能泄漏真实环境信息。

## 跨技术栈硬编码治理

任何技术栈只要触碰代码、配置、模板、测试或脚本，都要检查本次触碰范围内的硬编码值。先执行 `#跨技术栈编码风格智能化门禁`，再判断具体治理方式。判断依据是业务语义和变化方式，不是变量名前缀；`SOURCE_TYPE_*`、`STATUS_*`、`FORMAT_*`、`PLATFORM_*`、`DEFAULT_*` 只是示例，任何稳定状态、类型、来源、平台、协议、格式、动作、阶段、结果、阈值、时间窗、开关和外部系统参数都要检查。

先按以下顺序分类：

| 值类型 | 优先放置 | 典型例子 | 不适合 |
| --- | --- | --- | --- |
| 固定闭合集合 | 枚举、联合类型、常量对象、字典映射 | 状态、类型、来源、动作、平台、消息格式、业务模式 | 用户可新增、运行期可扩展的值 |
| 协议字段名或 key | 协议常量、契约常量、生成类型、SDK 常量 | JSON 字段名、模板参数名、header name、route name、event name、storage key | 固定业务取值、状态 code |
| 技术标准常量 | 语言/框架内置常量、集中常量对象 | HTTP method/status、media type、header name、MIME、日期格式、字符集 | 环境 URL、密钥、业务状态 |
| 环境或运维可变值 | 配置文件、环境变量、配置中心、类型化配置对象 | baseUrl、token、secret、超时、重试、线程池、TTL、阈值 | 固定业务枚举 |
| 运行期业务可维护值 | 数据字典、数据库表、规则系统、后台配置 | 可新增渠道、可维护分类、动态模板、租户差异策略 | 编译期固定集合 |
| 单点局部字面量 | 局部常量或直接字面量 | 只在一个短方法内使用且语义自解释的 `0`、`1`、空字符串、测试夹具 | 多处复用、跨层传递、参与分支判断 |

触发检查：

- 同一值或同类值出现两处以上。
- 值参与 `if/switch/match`、模板条件、路由、权限、状态流转、序列化、持久化、接口契约或跨系统调用。
- 值需要展示文案、反查、校验、排序、兼容旧 code 或映射外部系统。
- DTO/VO/API schema、OpenAPI/Swagger 注解、字段注释、接口文档或测试 fixture 已经列出固定 code/label 取值。
- 代码里出现平台编码、消息格式、content type、header、协议版本、外部渠道、webhook 类型、通知模板等外部协议字段。
- 本次修改虽然只碰一行，但该行所在方法已经有明显魔法字符串、魔法数字或重复常量。
- IDE 截图、diff、片段或调用链相关文件中已经能看到外部 URL、平台 code、消息格式、content type、超时、token key、配置 key 等被写死；即使用户只把截图当示例，也要在真实目标文件和直接调用链中做同类扫描。

处理原则：

- 先查项目是否已有枚举、常量模块、字典体系、配置类、i18n、design tokens、api client、SDK 常量或协议定义；有则复用，不重复新建。
- 已有枚举、联合类型、字典或生成类型能表达同一业务域时，实际赋值、比较、校验、默认值、mock 和序列化转换必须通过其 code/lookup/validator/类型收窄能力完成；即使公共 API 为兼容暂时保留 `String` 字段，业务代码也不能继续裸写同一组 code。
- 固定闭合集合优先使用本技术栈最自然的表达：Java `enum`，Python 3.11+ 可用 `StrEnum`，较低版本用 `Enum` + `str` mixin、`Enum` 或 `Literal`，TypeScript `as const` 对象、字符串联合类型或 `enum`（按项目风格），小程序原生使用集中 `constants`/字典对象。
- 技术标准值优先使用框架或标准库提供的常量；例如 HTTP status、Content-Type、MediaType、header name，不自己散写字符串。
- 环境差异、密钥、URL、超时、重试、阈值、开关、线程池、缓存 TTL 等不要写死在业务类或组件里。
- 如果抽取会改变 API、数据库 code、外部协议、序列化值或前端展示，必须保持兼容，必要时只先集中常量并说明后续迁移。
- 测试中允许保留少量直观字面量，但跨多个测试复用的状态、角色、协议、渠道、错误码和 fixture key 也应集中。
- 不为了“消灭所有字面量”制造过度抽象；真正的目标是让变化点集中、含义可跳转、错误更少。

## 跨技术栈重复逻辑治理

任何技术栈触碰代码时，都要检查本次触碰范围内是否存在字段不同但结构相同的重复逻辑。判断依据是“变化点是否稳定、重复是否会继续增加、抽象后是否更容易维护”，不是代码行数。

触发检查：

- 多个分支只差状态、类型、来源、平台、动作、格式、阈值、错误文案或目标字段。
- 多段赋值、映射、转换、默认值填充、校验、过滤、排序、权限判断、展示组装只有字段不同。
- 同一批字段在 Request/DTO/Entity/VO/ViewModel/Form/Excel/接口响应/mock/测试夹具之间反复手写转换。
- 同一业务规则在 Controller/Service/组件/store/hook/composable/script/test 中重复出现。
- 未来新增一种状态、渠道、消息格式、平台或外部系统时，预计要改多个位置。

优先选择当前技术栈已有能力：

- 类型和固定分支：枚举行为、联合类型、字典对象、策略表、handler map、状态机或多态。
- 字段映射：项目已有 mapper、converter、serializer、schema、form adapter、view model adapter 或测试 fixture builder。
- 校验：项目已有 validation/schema/form rules/annotation/decorator，而不是多处手写 `if return error`。
- 默认值和转换：集中 helper、工厂方法、构造器、builder、computed/selector、hook/composable、dataclass/model 方法。
- 配置驱动：稳定规则但值可运维调整时，走配置、字典、规则系统或配置中心。

边界：

- 两处以内、短小且语义完全不同的重复可以暂不抽象，但要避免继续复制。
- 抽象必须让业务意图更清楚；不要为了“高级”引入难读的反射、动态执行、复杂泛型、深层函数式链或过度继承。
- 性能敏感路径谨慎使用反射型拷贝、深拷贝和运行期动态分发；优先编译期映射、显式 helper 或项目已有高性能方案。
- 如果抽象会改变 API、序列化、数据库、权限、事务、前端契约或外部协议，先保持兼容并说明迁移方式。

## 跨技术栈注释原则

注释必须放在对应技术栈最能承载契约的位置，不统一堆在文件顶部，也不散落成逐行噪音。

- 通用：注释说明业务原因、边界、兼容性、风险、外部协议和非显然约束；不重复代码正在做什么。
- Java：Service 接口写契约，重要实现方法写原因和边界，Entity/DTO/VO 写关键字段语义，Controller 不用注释解释大段业务流程。
- Python：模块、公共类、公共函数和复杂算法用 docstring；复杂类型、配置对象、外部协议、数据清洗规则在定义处说明；普通步骤不逐行注释。
- Vue：组件用途、props/emits/slots 契约写在 `<script>` 对应定义附近；template 注释只保留非显然布局、权限、兼容或 slot 边界；style 注释只说明 token、覆盖第三方样式或兼容原因。
- React：组件和自定义 Hook 的用途写在定义或导出处；复杂 props 类型字段、副作用、memoization、ref/context 边界在对应定义附近说明；JSX 注释只保留非显然分支、可访问性和兼容逻辑。
- 小程序：页面/组件用途写在 `Page`、`Component`、Taro/Vue 组件定义或配置对象附近；`properties`、`observers`、生命周期、`setData` 边界、分包入口和平台条件编译在对应定义处说明；`wxml`/模板注释只保留非显然权限、slot、兼容和布局边界。
- SQL/Mapper：说明查询边界、性能原因、索引依赖、兼容字段和数据权限条件，不重复字段名。
- 配置文件：说明配置项来源、环境差异、安全边界和默认值原因，不写真实密钥或生产敏感信息。

新增、修改、删除、重构、阅读、检索、lint/format/typecheck 修复、截图审查、diff 审查、调用链确认和只做诊断时，只要已读到或为完成任务必须读取的文件中出现明确契约边界，都必须同步检查注释是否位于正确位置；缺失就补或写入当前任务胶囊评估，冗余就收敛。不能因为当前显性报错只是缩进、格式化或类型错误，就跳过同一语义单元内明显的注释契约缺口。

## 注释契约自动触发

注释规则不是只在用户要求“补注释”或代码新增时触发。以下信号一旦出现在触碰范围、直接调用链、截图、diff、lint/typecheck 输出定位文件、搜索结果打开文件或为完成任务必须读取的文件中，必须内部追加注释检查：

- 对外契约：导出的接口、类型、类、函数、Hook、composable、SDK/API client、Service 接口、请求/响应对象、DTO/VO/Entity、SQL/Mapper、配置项、事件名、路由元数据、权限/租户/审计字段。
- 前端 TypeScript 契约：`export interface`、`export type`、`Props`、`Emits`、`Slots`、`Request`、`Response`、`Params`、`Result`、`*Api` 命名空间或 `api/methods`、`service`、`request` 目录里的方法和类型。
- 业务含义字段：状态、类型、来源、阶段、结果、数量统计、时间字段、外部系统 id、兼容字段、展示字段、脱敏字段、权限字段、分页/筛选/排序参数。
- 非显然边界：loading、缓存、重试、幂等、提交中、错误处理、权限降级、兼容旧值、后端字典映射、外部协议枚举和单位换算。
- 工具暴露信号：ESLint、Prettier、Stylelint、TypeScript、Maven、pytest 等只报告了格式或类型位置，但打开的完整语义单元本身是对外契约且缺少必要说明。

处理顺序：

1. 先判断当前任务是否允许写入。允许写入且低风险闭环时，直接在契约定义处补最小必要注释，不另起全文件重构。
2. 如果用户只要求诊断、当前语义不闭环或补注释可能改变公共文档口径，先记录为强规则同步项和风险边界，说明需要的业务语义证据。
3. 注释只补“类型和命名无法表达”的信息：业务含义、外部协议、单位、取值来源、兼容原因、权限/脱敏边界、异常或副作用；不要给 `name`、`id`、`pageSize` 这类完全自解释且无额外业务语义的字段堆废话。
4. 先查项目已有注释风格、JSDoc/docstring/Javadoc、类型声明、OpenAPI/生成类型、README/Story/测试说明；有既有范式必须跟随，不创造并行格式。
5. 对 lint/format 修复，先修格式，再在同一触碰语义单元做注释契约扫描；只处理当前文件或直接调用链内低风险缺口，历史全仓缺口记录边界。

## 高风险变更

涉及 API、DTO、SQL、数据库字段、权限、状态管理、枚举或业务规则时：

1. 说明修改原因。
2. 列出影响范围。
3. 优先兼容旧逻辑。
4. 验证完整流程。

## 项目演进规则

项目结构、模块边界、核心业务流程或技术架构发生明显变化时，需要给出 AGENTS 更新建议；用户同意后再修改 AGENTS。

重大变化包括新增一级模块、微服务、独立前端应用、AI 能力模块、数据库或存储组件、核心业务流程、公共 SDK、跨模块共享能力、统一网关或认证体系、大规模目录重构或重要架构调整。

新增规则必须向后兼容、避免重复、保持简洁，只补充增量内容。普通业务代码变更不要频繁修改 AGENTS。
