# 全局工程规则

本文件只保留所有编程任务都需要先看的通用规则。各技术栈 reference 只保留落地差异；文件归属、环境命令、验证、安全边界、硬编码、重复逻辑和注释先按本文件判断，再进入对应技术栈文件。

## 语言偏好

- 默认使用简体中文回复。
- 分析、解释、计划、变更说明、验证结果、根因分析、下一步建议都用简体中文。
- 代码、命令、配置项、环境变量、文件名、类名、方法名、日志、异常信息保持原文。
- 不使用英文标题 `What Changed`、`Validation`、`Root Cause`、`Next Steps`、`Summary`。
- 使用中文标题：`变更内容`、`验证结果`、`根因分析`、`下一步`、`总结`。

## 工具优先级

JetBrains 项目必须优先使用 JetBrains MCP / IDE 工具完成代码读取、定位、修改和诊断。

- 只有 JetBrains MCP 明确不可用、超时或返回错误时，才允许使用 Shell。
- 禁止未尝试 JetBrains MCP 就直接用 Shell 扫描或修改 JetBrains 项目。
- 优先分析最小范围，不做无意义全仓扫描。
- 未明确要求时不要使用 Sub Agent。

非 JetBrains 项目可直接使用本地文件工具。搜索优先 `rg` 或 `rg --files`。

## 修改前确认

修改前必须确认：

1. 目标文件。
2. 问题根因。
3. 最小修改方案。
4. 不应受影响的模块、接口、数据结构、权限和业务逻辑。

修改前再次引用任务编号确认目标。若任务描述与实现方向不一致，立即停止并汇报。

## 核心规则

- 优先最小、安全、可回滚的修改。
- 除非任务明确要求，否则保持现有行为不变。
- 不修改无关文件，不随意重构正常工作的代码。
- 最小改动不是“只改新增行”或“只记风险”的豁免理由；当本次触碰范围、直接调用链或为完成任务必须读取的相关文件已经命中硬编码、重复逻辑、配置写死、分层错位、注释缺失、安全边界或验证缺口等明确治理项时，先判断调用链深度、涉及文件数量、契约风险和验证路径。影响面浅、文件少、兼容性明确时，必须写入当前任务清单和任务胶囊并直接做局部对齐；风险高或语义不闭环时才保留兼容并说明后续项。
- 不随意修改 API、DTO、数据库字段、权限、路由、配置键和公共契约。
- 未明确批准不要新增依赖。
- 未批准不要执行数据迁移、删除操作、网络写入和影响工作区外部数据的命令。
- 参考类似代码风格以及文件存放位置；新建文件归属地规则见 `07-java-backend-architecture.md#新建文件归属地`。

## 通用质量

- 优先复用现有模式，避免硬编码，保持接口清晰，保留未来扩展点。
- 做好输入校验，保留认证、权限、租户隔离和审计逻辑。
- 处理空值、异常值和边界情况，不吞异常。
- 避免 N+1 查询、全表扫描和无分页查询。
- 保持模块职责单一，优先清晰命名。
- 复杂逻辑说明原因而不是描述代码行为。

## 跨技术栈文件归属与依赖边界

任何技术栈新增、移动或拆分文件前，都要先确认“归属地”，再动手写文件。不要因为当前目录打开方便、文件名相近或功能临时可用，就把代码放到 root、错误 module、构建产物目录或依赖方向错误的位置。

通用确认项：

1. 目标文件属于哪个业务域或技术域。
2. 它是源码、测试、配置、脚本、模板、迁移、资源，还是构建产物。
3. 它属于哪个 module/package/workspace/app/subpackage。
4. 同类文件当前放在哪里，命名和导入方式如何。
5. 新文件会被谁依赖，它又依赖谁，是否造成反向依赖、循环依赖或跨层访问。
6. 生成目录和源码目录是否已分清，例如 `dist/`、`build/`、`target/`、`unpackage/dist/`、`miniprogram_npm/`、`generated/`。

处理原则：

- 先查同业务域、同后缀、同职责文件；优先放到既有模式下，不新造并行目录。
- 接口/契约、实现、实体、配置、脚本、测试和样例可能分属不同 module；不能只按文件名相似放置。
- 测试跟随被测对象所在模块和项目已有测试目录；不要为了运行方便放到聚合 root 或生产包里。
- 构建产物默认只读不改，除非任务明确要求分析产物或修复生成结果；修复应回到源码和构建配置。
- 归属不确定时，先列出候选位置和依据；不能确认依赖方向时不直接新建。

## 跨技术栈环境与命令

运行、构建、测试和安装依赖前，先按项目声明选择工具链，不凭个人机器习惯写命令。

通用顺序：

1. 读取项目缓存、README、脚本、CI、IDE/项目配置、版本文件和 lockfile。
2. 使用项目声明的运行时、包管理器、构建工具和 workspace/filter 方式。
3. 本机路径未知时按 `06-environment-discovery.md` 发现、验证并缓存到 `.codex/local-environment.json`。
4. 缓存与项目配置冲突时，以项目配置为准并重新验证。

约束：

- 不混用包管理器、解释器、JDK、Maven、Node、Python、模拟器或 CLI。
- 不为了临时通过而全局安装依赖、修改用户 shell 配置、修改 IDE 全局配置或写死个人绝对路径。
- 构建命令必须在正确 root/workspace/module 运行；多模块和多包项目先确认聚合 root 与目标模块。
- 命令输出只保留与任务相关的结论、错误、路径和版本，不搬运大段日志。

## 跨技术栈验证策略

验证要和触碰范围、风险等级、项目已有命令匹配。不要假验证，也不要为了一个局部修改无差别跑昂贵全量流程。

优先级：

1. 语法、类型、导入、配置可解析等最快失败检查。
2. 触碰文件或触碰方法的定向测试。
3. 直接调用链所在模块、package、workspace 或页面的测试。
4. 改公共契约、配置、路由、构建、权限、数据库、状态流转、并发或跨端能力时，再扩大到相关模块构建或集成验证。
5. UI、浏览器、小程序模拟器、真机、外部系统和发布链路只在触碰相应风险时执行或明确说明无法执行原因。

要求：

- 优先使用项目已有 `test`、`lint`、`typecheck`、`build`、CI、Makefile、tox/nox、workspace script 或框架命令。
- 无法运行验证时，说明缺少的环境、账号、密钥、模拟器、服务、数据或权限，不用静态推测冒充通过。
- 修改 bug 时优先补或运行能复现问题的定向验证；没有测试能力时至少给出最小复现命令或人工验收点。
- 验证失败先区分本次改动、历史失败、环境缺失和命令选择错误，不连续盲改。

## 跨技术栈安全与外部边界

涉及认证、授权、租户、审计、密钥、外部 API、文件、网络、支付、消息、上传下载、富文本、动态执行或生产配置时，默认按高风险边界处理。

通用规则：

- 不把密钥、token、账号、生产 URL、上传凭证、私钥、机器人编号、白名单和真实 appid 写进代码、测试、README 或日志。
- 不绕过认证、授权、数据权限、租户隔离、审计、脱敏、限流和重复提交防护。
- 外部调用必须有超时、错误处理、重试边界、幂等或去重策略；不能无限重试或吞异常返回成功。
- 文件路径、上传文件、压缩包、模板渲染、SQL、命令参数、动态 URL、富文本、反序列化和动态执行要做输入边界检查。
- 不在事务内直接执行不可回滚副作用；需要提交后动作时使用事务事件、afterCommit、outbox、可靠消息或项目已有机制。
- 示例、测试和 mock 可以使用假值，但要明显不是可用凭证，且不能泄漏真实环境信息。

## 跨技术栈硬编码治理

任何技术栈只要触碰代码、配置、模板、测试或脚本，都要检查本次触碰范围内的硬编码值。判断依据是业务语义和变化方式，不是变量名前缀；`SOURCE_TYPE_*`、`STATUS_*`、`FORMAT_*`、`PLATFORM_*`、`DEFAULT_*` 只是示例，任何稳定状态、类型、来源、平台、协议、格式、动作、阶段、结果、阈值、时间窗、开关和外部系统参数都要检查。

先按以下顺序分类：

| 值类型 | 优先放置 | 典型例子 | 不适合 |
| --- | --- | --- | --- |
| 固定闭合集合 | 枚举、联合类型、常量对象、字典映射 | 状态、类型、来源、动作、平台、消息格式、业务模式 | 用户可新增、运行期可扩展的值 |
| 技术标准常量 | 语言/框架内置常量、集中常量对象 | HTTP method/status、media type、header name、MIME、日期格式、字符集 | 环境 URL、密钥、业务状态 |
| 环境或运维可变值 | 配置文件、环境变量、配置中心、类型化配置对象 | baseUrl、token、secret、超时、重试、线程池、TTL、阈值 | 固定业务枚举 |
| 运行期业务可维护值 | 数据字典、数据库表、规则系统、后台配置 | 可新增渠道、可维护分类、动态模板、租户差异策略 | 编译期固定集合 |
| 单点局部字面量 | 局部常量或直接字面量 | 只在一个短方法内使用且语义自解释的 `0`、`1`、空字符串、测试夹具 | 多处复用、跨层传递、参与分支判断 |

触发检查：

- 同一值或同类值出现两处以上。
- 值参与 `if/switch/match`、模板条件、路由、权限、状态流转、序列化、持久化、接口契约或跨系统调用。
- 值需要展示文案、反查、校验、排序、兼容旧 code 或映射外部系统。
- 代码里出现平台编码、消息格式、content type、header、协议版本、外部渠道、webhook 类型、通知模板等外部协议字段。
- 本次修改虽然只碰一行，但该行所在方法已经有明显魔法字符串、魔法数字或重复常量。
- IDE 截图、diff、片段或调用链相关文件中已经能看到外部 URL、平台 code、消息格式、content type、超时、token key、配置 key 等被写死；即使用户只把截图当示例，也要在真实目标文件和直接调用链中做同类扫描。

处理原则：

- 先查项目是否已有枚举、常量模块、字典体系、配置类、i18n、design tokens、api client、SDK 常量或协议定义；有则复用，不重复新建。
- 固定闭合集合优先使用本技术栈最自然的表达：Java `enum`，Python `Enum/StrEnum/Literal`，TypeScript `as const` 对象、字符串联合类型或 `enum`（按项目风格），小程序原生使用集中 `constants`/字典对象。
- 技术标准值优先使用框架或标准库提供的常量；例如 HTTP status、Content-Type、MediaType、header name，不自己散写字符串。
- 环境差异、密钥、URL、超时、重试、阈值、开关、线程池、缓存 TTL 等不要写死在业务类或组件里。
- 如果抽取会改变 API、数据库 code、外部协议、序列化值或前端展示，必须保持兼容，必要时只先集中常量并说明后续迁移。
- 测试中允许保留少量直观字面量，但跨多个测试复用的状态、角色、协议、渠道、错误码和 fixture key 也应集中。
- 不为了“消灭所有字面量”制造过度抽象；真正的目标是让变化点集中、含义可跳转、错误更少。

## 跨技术栈重复逻辑治理

任何技术栈触碰代码时，都要检查本次触碰范围内是否存在字段不同但结构相同的重复逻辑。判断依据是“变化点是否稳定、重复是否会继续增加、抽象后是否更容易维护”，不是代码行数。

触发检查：

- 多个分支只差状态、类型、来源、平台、动作、格式、阈值、错误文案或目标字段。
- 多段赋值、映射、转换、默认值填充、校验、过滤、排序、权限判断、展示组装只有字段不同。
- 同一批字段在 Request/DTO/Entity/VO/ViewModel/Form/Excel/接口响应/mock/测试夹具之间反复手写转换。
- 同一业务规则在 Controller/Service/组件/store/hook/composable/script/test 中重复出现。
- 未来新增一种状态、渠道、消息格式、平台或外部系统时，预计要改多个位置。

优先选择当前技术栈已有能力：

- 类型和固定分支：枚举行为、联合类型、字典对象、策略表、handler map、状态机或多态。
- 字段映射：项目已有 mapper、converter、serializer、schema、form adapter、view model adapter 或测试 fixture builder。
- 校验：项目已有 validation/schema/form rules/annotation/decorator，而不是多处手写 `if return error`。
- 默认值和转换：集中 helper、工厂方法、构造器、builder、computed/selector、hook/composable、dataclass/model 方法。
- 配置驱动：稳定规则但值可运维调整时，走配置、字典、规则系统或配置中心。

边界：

- 两处以内、短小且语义完全不同的重复可以暂不抽象，但要避免继续复制。
- 抽象必须让业务意图更清楚；不要为了“高级”引入难读的反射、动态执行、复杂泛型、深层函数式链或过度继承。
- 性能敏感路径谨慎使用反射型拷贝、深拷贝和运行期动态分发；优先编译期映射、显式 helper 或项目已有高性能方案。
- 如果抽象会改变 API、序列化、数据库、权限、事务、前端契约或外部协议，先保持兼容并说明迁移方式。

## 跨技术栈注释原则

注释必须放在对应技术栈最能承载契约的位置，不统一堆在文件顶部，也不散落成逐行噪音。

- 通用：注释说明业务原因、边界、兼容性、风险、外部协议和非显然约束；不重复代码正在做什么。
- Java：Service 接口写契约，重要实现方法写原因和边界，Entity/DTO/VO 写关键字段语义，Controller 不用注释解释大段业务流程。
- Python：模块、公共类、公共函数和复杂算法用 docstring；复杂类型、配置对象、外部协议、数据清洗规则在定义处说明；普通步骤不逐行注释。
- Vue：组件用途、props/emits/slots 契约写在 `<script>` 对应定义附近；template 注释只保留非显然布局、权限、兼容或 slot 边界；style 注释只说明 token、覆盖第三方样式或兼容原因。
- React：组件和自定义 Hook 的用途写在定义或导出处；复杂 props 类型字段、副作用、memoization、ref/context 边界在对应定义附近说明；JSX 注释只保留非显然分支、可访问性和兼容逻辑。
- 小程序：页面/组件用途写在 `Page`、`Component`、Taro/Vue 组件定义或配置对象附近；`properties`、`observers`、生命周期、`setData` 边界、分包入口和平台条件编译在对应定义处说明；`wxml`/模板注释只保留非显然权限、slot、兼容和布局边界。
- SQL/Mapper：说明查询边界、性能原因、索引依赖、兼容字段和数据权限条件，不重复字段名。
- 配置文件：说明配置项来源、环境差异、安全边界和默认值原因，不写真实密钥或生产敏感信息。

新增或修改代码时，触碰到接口、实体、DTO/VO、组件 props/slots/children、hook/composable、配置项、SQL 或复杂业务方法，必须同步检查注释是否位于正确位置；缺失就补，冗余就收敛。

## 高风险变更

涉及 API、DTO、SQL、数据库字段、权限、状态管理、枚举或业务规则时：

1. 说明修改原因。
2. 列出影响范围。
3. 优先兼容旧逻辑。
4. 验证完整流程。

## 项目演进规则

项目结构、模块边界、核心业务流程或技术架构发生明显变化时，需要给出 AGENTS 更新建议；用户同意后再修改 AGENTS。

重大变化包括新增一级模块、微服务、独立前端应用、AI 能力模块、数据库或存储组件、核心业务流程、公共 SDK、跨模块共享能力、统一网关或认证体系、大规模目录重构或重要架构调整。

新增规则必须向后兼容、避免重复、保持简洁，只补充增量内容。普通业务代码变更不要频繁修改 AGENTS。
