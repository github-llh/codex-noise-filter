# 并发、异步与批量处理

本文件按需读取。是否打开由本 skill 根据代码证据、调用链、事务/副作用边界、任务调度、消息/线程/批量入口和风险信号内部判定；命中高并发、幂等、锁、死锁、异步事件、中间件、批量操作、线程池、虚拟线程或用户上下文传播时必须打开，外部点名不作为前提。外部副作用、安全边界、环境命令和验证策略先按 `01-global-engineering-rules.md` 判断；本文件只补并发、异步和批量落地细则。

## 高并发可用性

先执行 `01-global-engineering-rules.md#跨技术栈安全与外部边界` 和 `01-global-engineering-rules.md#跨技术栈验证策略`，再确认并发控制方案。

高并发写代码前先确认：

1. 共享资源是什么：数据库行、库存/额度、状态、缓存 key、文件、消息、外部接口。
2. 是否允许重复请求、乱序请求、并发提交、超时重试。
3. 一致性要求是强一致、最终一致，还是可补偿。
4. 当前项目已有锁、幂等、限流、队列、缓存、消息中间件、任务调度和重试机制。

优先复用现有技术栈：

- 数据库唯一索引、乐观锁、悲观锁、条件更新、版本号、状态机前置条件。
- Redis 分布式锁、去重 key、限流、缓存穿透/击穿/雪崩防护。
- 消息队列、事件总线、任务表、outbox、延迟队列、重试队列。
- 项目已有线程池、异步执行器、任务调度器、Trace/UserContext 传播工具。

不要为了并发直接新增中间件或线程池；先查项目已有能力。

## 幂等性

涉及创建、支付、审核、状态流转、文件生成、消息消费、外部回调、批量导入时，默认考虑幂等。

常用方式：

- 业务唯一键或请求幂等 key。
- 数据库唯一索引兜底。
- 状态机条件更新，例如 `where id = ? and status = ?`。
- 消息消费表、处理流水表、去重表。
- Redis 短期去重 key，但关键业务不能只依赖缓存。

幂等结果要明确：重复请求返回已处理结果、忽略、报业务错误，还是触发补偿。

## 死锁规避

写多表、多行、批量更新前必须考虑死锁。

- 多处代码更新相同资源时，保持一致加锁和更新顺序。
- 批量 ID 先排序，再按固定顺序更新。
- 控制事务范围，避免长事务和事务内远程调用。
- 避免大批量一次性锁太多行；分批提交并控制批次大小。
- 使用索引命中更新条件，避免无索引更新导致范围锁扩大。
- 捕获数据库死锁异常时，按项目已有策略重试；重试必须有次数、退避和幂等保证。

## 异步事件与中间件

可以异步化的场景：

- 耗时但不影响主流程成功的通知、日志、报表生成、文件转换、缓存预热、外部同步。
- 可最终一致、可重试、可补偿的副作用。

不应直接异步化的场景：

- 需要同步返回强一致结果的核心写操作。
- 权限校验、金额/库存扣减、状态前置校验等主事务必要步骤。

事务配合：

- 事务提交后再发消息或事件，优先使用事务同步回调、事务事件、outbox 或项目已有可靠消息机制。
- 消费端必须幂等，能处理重复消息、乱序消息和延迟消息。
- 消息体要包含业务主键、幂等键、操作者、租户、追踪 ID 和必要上下文，不依赖消费线程从当前请求上下文读取。

## 批量操作与并发执行

批量处理先确认 Java 版本和项目已有并发设施：

- 查看 `pom.xml`、compiler release/source/target、IDE SDK，确认是否支持虚拟线程。
- 优先复用项目已有 `TaskExecutor`、线程池、调度器、批处理框架和监控配置。
- Java 21+ 且项目已允许时，可评估虚拟线程处理大量 IO 型任务；CPU 密集任务不因虚拟线程自动变快。

批量规则：

- 先分批，再考虑并发；每批大小按数据库、接口、内存和锁范围控制。
- 并发数必须有上限，不能无界 `parallelStream` 或无限提交任务。
- 每个任务必须有超时、失败收集、重试/补偿策略和可观测日志。
- 批量写库要避免长事务；必要时按批次建立事务。
- 批量读写大数据要分页、游标或流式处理，避免一次性加载全量。

## 用户上下文传播

异步、线程池、虚拟线程、消息消费中不能假设能从当前请求线程拿到用户信息。

必须处理：

- 创建人、修改人、租户、部门、数据权限、审计字段、TraceId。
- 异步任务需要的用户 ID、租户 ID、权限范围应显式传入任务参数或消息体。
- 如果当前项目有 `UserContext`、`TenantContext`、`SecurityContext`、`MDC` 传播工具，优先复用。
- 消费端从消息体恢复必要上下文，处理完成后清理 ThreadLocal/MDC，避免串号。
- 无法获得真实用户时，使用项目约定的系统用户，并在审计中标明来源。

## 项目索引性能

涉及并发、批量和异步改造时，定位要先走索引：

- 先查现有 executor、scheduler、mq、event、lock、idempotent、tenant、user context、audit 相关类。
- 只读同类实现和调用链，不全仓扫描。
- 优先复用已有组件，避免新增平行体系导致维护成本和索引噪音上升。
