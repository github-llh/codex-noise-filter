# 交付模板

## 最终回复结构

默认用简体中文，短任务可合并为一两段。复杂任务建议使用：

**变更内容**

- 列出修改文件。
- 说明修改目的。
- 说明影响范围。

**验证结果**

- 写明执行过的最轻量验证命令。
- 写明通过、失败或未执行原因。

**风险与边界**

- 说明未改动的模块、接口、数据结构、权限和业务逻辑。
- 高风险变更要写回滚点。

不要把工具原始大段日志贴进最终回复，只摘要关键结论。

## 任务胶囊模板

```text
目标：...
允许改动：...
禁止改动：...
强规则同步项：...（触碰范围/直接调用链/必要读取文件内，低风险闭环则本轮处理；高风险则写边界）
编码风格预检：...（逐项分类：协议 key / 技术常量 / 固定业务取值 / 配置值 / 运行期字典值 / 局部字面量；常量/枚举/配置/字典归属；重复分支；允许保留原因）
预期产物：...
读取预算：最多 ... 个文件，每个默认 ... 行窗口。
```

## 修改前确认模板

```text
任务 #... 修改前确认：
目标文件：...
根因：...
最小方案：...
不影响：模块/API/DTO/数据库/权限/业务逻辑 ...
```

## 上下文胶囊

Context Capsule 是自动恢复锚点，不是只在 Plan 模式或手工切换窗口时才写。只要任务进入复杂读取、修改、验证、失败回退、自动续跑、上下文恢复或新指令插入，就必须维护最新胶囊；接近上下文阈值、长时间工具执行前后、跨窗口、任务被打断或模型可能自动压缩时，必须输出可恢复版本。

压缩友好版本控制在 12 到 16 行，优先保留不可从当前窗口稳定恢复的信息：

```text
Context Capsule
目标/阶段：...（定位/验证/修改/复核/失败回退）
任务清单：#1 ...；#2 ...（完成/进行中/待验证）
允许/禁止：允许改 ...；禁止改 API/DTO/DB/权限/业务语义 ...
规则：已读 SKILL/00/02/05/...；仍需追加 ...
触碰范围：文件/符号 ...；不触碰 ...
调用链：已闭环 ...；缺口 ...
强规则同步项：低风险本轮处理 ...；高风险边界 ...
风格预检：新增/保留字面量分类 ...；常量/枚举/配置归属 ...
读取/历史：完整语义单元 ...；未读边界 ...；git 历史结论 ...
失败/策略：失败类型 ...；已写入/未写入 ...；切换策略 ...
验证：已跑 ...；未跑原因 ...；仍需验证 ...
证据：文件路径:行号 - 事实；命令 - 结论
已改/回滚：已改 ...；回滚文件/补丁 ...
下一步：只做一个最小闭环 ...
```

继续操作前，优先从当前会话、Context Capsule、git diff、已读证据、任务清单和当前文件重新读取结果自动重建边界；只有无法完整重建任务边界、约束状态、证据锚点、已写入状态和回滚清单时，才要求补充或确认，例如 `按 Capsule 继续`。边界待补全时不直接进入修改。

## Codex 上下文管理

用于 Codex 长任务、跨窗口、自动压缩、自动续跑、失败回退或新指令中途插入的场景。默认假设窗口会频繁压缩，不能依赖“模型还记得”；重要状态必须进入任务胶囊或 Context Capsule。

### 自动维护

- 每完成一个关键阶段，自动更新任务状态：定位、验证、修改、复核、失败回退。
- 每次修改前后、验证前后、失败后、策略切换后、新指令插入后，都要确认 Capsule 是否仍覆盖最新任务清单、触碰范围、调用链、强规则同步项、失败状态、验证状态和回滚点。
- 当输出即将变长、工具可能长时间运行、连续读取多个文件、准备批量修改、出现补丁失败或准备结束当前轮时，先给出或刷新压缩版 Capsule。
- 已确认的调用链、约束、不变项、读取完整性、git 历史结论、已写入/未写入状态和回滚点不得在后续回复中丢失。

### 保留优先级

必须保留：

- 原始目标、当前阶段、任务清单状态和用户最新约束。
- 允许改动和禁止改动，尤其是 API、DTO、数据库、权限、路由、配置键、外部协议和业务语义边界。
- 已读 reference、触碰范围、直接调用链、影响面、未闭环缺口和回滚点。
- 强规则同步项、低风险本轮处理项、高风险不处理边界。
- 编码风格预检结论：新增/保留字面量分类、常量/枚举/配置/字典归属、重复逻辑抽象边界。
- 读取完整性、git 历史结论、失败类型、策略切换、已写入/未写入文件、验证命令和验证结果。
- 证据锚点，格式使用 `文件路径:行号 - 事实` 或 `命令 - 结论`。

可以丢弃或摘要：

- 大段工具输出、重复日志、完整文件正文、已被 diff 覆盖的中间草稿、已失效假设、和当前闭环无关的搜索结果。
- 已经写入文件且可由 `git diff` 恢复的长文本，只保留文件路径、语义摘要和回滚方式。

### 压缩恢复

- 自动压缩、跨窗口、新会话或自动续跑后的第一步，先读取 `SKILL.md`、`references/00-index.md` 和当前任务命中的 reference，再用 Capsule、`git diff`、当前文件原文和已读证据恢复任务边界。
- 恢复后必须核对：用户最新指令是否改变目标；任务清单是否完整；触碰范围和禁止项是否仍成立；调用链/强规则同步项/失败策略/验证状态是否缺失。
- Capsule 缺少关键证据、回滚信息、已写入状态或策略切换依据时，先补读当前文件和 diff；仍无法恢复时，只问一个最小澄清问题，不直接继续写代码。
- 不允许只写“延续上文”或凭旧记忆继续；压缩后所有继续修改都必须以当前文件和最新规则重新校准。

### 新目标插入

- 新目标默认作为增量任务，不覆盖原任务。
- 先判断关系：继续主任务、并行执行、暂时转向、替代执行。
- 若新目标会改变 API、DTO、权限、数据库、构建配置或业务规则，先停下说明影响范围。
- 处理完插入目标后，更新 Context Capsule，保证主任务可恢复。

### 全局 AGENTS 的定位

- 本 skill 已内化全局 `AGENTS.md` 中应保留的编程规则，编程任务优先读 skill 索引。
- 全局 `AGENTS.md` 仍建议保留，作为非 skill 场景、未触发 skill 场景和其他 agent 的兜底规则。
- 不建议删除全局 `AGENTS.md`；如要瘦身，可保留语言偏好、工具优先级和“编程任务启用 codex-noise-filter”的指向。

## Codex 记忆管理

Codex 的记忆分三层处理：

1. 会话内上下文：用 Context Capsule 保存当前任务边界、证据、已改、回滚和下一步。
2. 工作区本地缓存：用 `.codex/local-environment.json` 保存已验证的环境路径，例如 Maven、JDK、Node、包管理器；该文件只服务当前工作区，默认不提交。
3. 长期 memory：只记录稳定偏好和跨任务复用规则；机器私有绝对路径、临时错误日志、一次性构建结果和未验证猜测不要写入长期记忆。

### 何时读取

- 新任务涉及当前仓库、skill、构建工具、IDE 配置或用户历史偏好时，先做轻量 memory pass。
- 如果只是简单命令、翻译、纯问答，不读长期 memory。
- 若 memory 与当前项目配置冲突，以当前项目配置和已验证本地缓存为准。

### 何时缓存

- 只有路径存在且通过最小验证后才缓存。
- 缓存项必须记录来源、验证命令、验证时间和适用范围。
- 长期 memory 只在存在明确记忆更新授权时写入；否则只写工作区本地缓存或在回复中说明。

### 何时失效

- 可执行文件不存在、版本验证失败、项目配置显式指定不同路径时，缓存失效。
- 缓存失效后重新走发现流程，不继续使用旧路径。

## 常用短语

- `先做调用链确认，再决定改动。`
- `当前读取范围已收敛到 X 个文件，预估新增 token 约 Y。`
- `改动前调用链未闭环，先补充。`
- `这一步只做最小闭环修改，不扩展到无关模块。`
