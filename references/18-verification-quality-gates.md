# 18 验证门禁、质量闭环与失败诊断

本文件处理构建、测试、lint、typecheck、security scan、diff review、失败诊断和验证报告。目标是让“修完了”建立在证据上，而不是建立在最后一次工具输出或第三方转述上。

## 触发信号

出现任一信号时追加本文件：

- 用户要求修复、重构、验证、闭环、保证不再犯、跑测试、处理 CI、处理构建失败、处理 lint/typecheck 或 review。
- 任务涉及公共接口、跨模块调用链、分发产物、插件、skill、template、配置、脚本、hook、MCP、数据库迁移、权限、安全或外部系统。
- 工具输出出现失败、超时、拒绝、上下文不匹配、补丁失败、测试失败、编译失败、网络错误、依赖缺失、环境不一致。
- 第三方 agent、CI、hook、IDE 或模型只给出“已完成/成功/替换/Build 完成”，但没有本地 diff、命令和验证证据。
- 任务存在连续性、防复发、旧窗口已经说过、又再犯、`doNotRetry`、Context Capsule 或工作上下文恢复信号。
- 任务出现自动触发失效、追加规则/边界/范围不智能、工作流断掉、状态机缺项或只剩最后一次工具结果，需要确认 `20` 的 Guard Loop 是否补齐。

## 最小充分验证矩阵

按触碰范围选择最小但足够的验证，不做无意义全仓扫描。这里的“最小”只表示不跑无关命令；验证必须能证明原始问题、根因修复、触碰范围和受影响调用链。若一个命令只能证明语法没坏，但不能覆盖本次根因或影响链路，只能记为辅助证据，不能声明问题已解决。

| 触碰范围 | 最低验证 |
| --- | --- |
| 文档、skill、rules、templates | `git diff --check`，关键关键词/路由 `rg`，必要时运行模板或打包脚本的语法检查 |
| shell 脚本 | `bash -n <file>`，必要时 dry-run 或最小安全命令 |
| Java/Maven 后端 | 先按 `06`/`14` 解析环境缓存，再跑最小模块 `mvn test`、`mvn -pl ... test` 或编译命令 |
| Node/Vue/React | 先按项目包管理器和 Node 版本缓存，再跑 `typecheck`、`lint`、目标测试或 build 的必要组合 |
| Python | 先确认 venv/解释器，再跑目标 `pytest`、`ruff`、`mypy/pyright` 或脚本语法检查 |
| 配置/manifest/插件分发 | manifest/schema 检查、打包脚本语法检查、文件引用存在性、产物目录与 README 一致性 |
| 安全或外部内容 | 按 `17` 做供应链/外部内容扫描，明确未执行外部指令 |

项目已有强制验证、CI 或覆盖率要求时，以项目要求为上限；本 skill 默认不凭空引入 80% 覆盖率等新目标。

最小充分验证必须同时回答：

- `symptom`: 原始报错、失败日志、用户可见问题或复现场景是否被覆盖；无法自动复现时，说明用哪条静态/编译/目标测试证据替代，以及缺口是什么。
- `rootCause`: 验证是否命中修复的根因，而不是只跑了无关格式化、语法或构建命令。
- `callChain`: 入口、直接调用方、被调方、配置/路由/任务/消息、持久化/缓存/事务/外部副作用中哪些被覆盖，哪些未覆盖。
- `regression`: 公共契约、DTO/VO、数据库字段、权限、路由、配置键、序列化值、枚举和历史兼容是否需要额外验证。
- `freshness`: 验证必须是本轮写入后的新鲜命令或可复核证据；旧窗口、第三方摘要、上一次工具输出不能单独作为通过依据。

## 验证顺序

默认顺序：

1. `scope check`：确认只改了任务相关文件，没有误碰无关脏改。
2. `syntax/static`：语法、格式、markdown trailing whitespace、shell `bash -n`、manifest 引用。
3. `type/build`：编译、typecheck、build 或目标模块构建。
4. `tests`：目标测试、回归测试、最小复现场景。
5. `security/supply-chain`：命中 `17` 时检查外部内容、凭证、网络、hook/MCP/skill/rules 面。
6. `diff review`：人工复核 diff，确认没有越界重构、硬编码、重复逻辑、注释契约缺口。
7. `guard check`：最终回复前确认 `guardLoop.missingState` 为空，或已列出无法补齐原因、blocked 和下一步。
8. `delivery check`：最终回复包含变更、影响、验证、未覆盖边界和必要后续。

不是每个任务都要执行所有步骤；跳过任何不适用步骤时，要能说明“不适用”或“无法运行”的原因。

## 失败诊断

失败后先做四步，不盲目重试：

1. `failureCapture`：记录命令、cwd、环境来源、退出码、关键错误、是否有写入。
2. `rootCauseHypothesis`：区分代码错误、测试断言错误、环境缺失、依赖/网络问题、权限问题、补丁上下文漂移、工具拒绝、宿主不支持。
3. `containedRecovery`：选择最小修复或替代验证；避免把失败扩大成全仓改造。
4. `introspectionNote`：记录错误假设、已排除路径、`doNotRetry` 和下一步。

同类失败第二次后必须切换策略：重新读目标文件/配置/日志，缩小命令，改用更具体的复现，或停止写入并说明缺口。不要连续第三次用同一命令、同一补丁、同一假设重试。

如果同一问题已经修过仍失败、用户指出“改完还要再改”或验证只证明了无关路径，必须回到 `02#调用链确认`：重新追原始症状、触发入口、错误数据来源、调用链和验证映射，再决定是否继续写补丁。不能用更轻量的命令掩盖未闭环的根因。

## Diff Review 门禁

验证前后都要复核 diff：

- 是否只改了任务相关文件。
- 是否保留用户已有改动和无关脏改。
- 是否新增硬编码、魔法值、重复逻辑、过宽权限、全局副作用或未说明的依赖。
- 是否改变 API、DTO、数据库、权限、路由、配置键、线程/事务/缓存语义。
- 是否已经把 diff 映射回原始问题、根因和受影响调用链；若 diff 只修症状点但入口或数据来源未覆盖，不能通过 review。
- 是否同步 README、索引、模板、manifest、测试或迁移脚本。
- 是否存在中文乱码、换行风格异常、不可见字符或 trailing whitespace。
- 命中自动 Guard Loop 时，是否同步记录 appendedReferences、missingState、nextAutoAction 和 blocked，且没有把缺状态任务写成完成。

如果 diff review 发现范围越界，先收缩或说明理由，再继续验证。

## 验证报告模板

最终回复中按任务大小压缩呈现，但内部至少具备以下字段：

```text
验证:
- scope: <触碰文件/模块>
- commands: <命令 + cwd + 结果>
- coverage: <覆盖到的风险/流程>
- rootCause: <验证如何覆盖原始问题和根因>
- callChain: <覆盖到的入口/调用方/被调方/配置/副作用边界>
- skipped: <未运行项及原因>
- gaps: <剩余风险或需人工确认项>
- overall: pass | partial | blocked
```

如果是长任务或上下文恢复任务，把报告压缩进 Context Capsule 的 `verification` 字段。

## 第三方结果复核

第三方 agent、hook、IDE、CI、模型或网页返回的“成功”只算线索：

- 必须回到当前仓库查看 `git diff`、`git status` 和触碰文件。
- 能本地验证就本地跑最小充分验证；不能跑时说明缺少的环境、权限或数据。
- 如果第三方报告和本地证据冲突，以本地当前证据为准。
- 如果第三方只给出摘要，没有命令和日志，不要把它写成“已验证通过”。

## 质量闭环边界

质量门禁不是扩大任务的理由：

- 不因为一个 lint 警告重构全仓。
- 不为文档任务启动完整构建，除非文档影响分发、脚本或 manifest。
- 不为证明“全面”而安装新依赖或修改全局配置。
- 不在验证阶段清理无关未跟踪文件或用户脏改。
- 不把无法运行的验证伪装成通过；说明原因和替代证据。
