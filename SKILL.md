# codex-noise-filter

目的：
- 在保持任务完成质量的前提下，显著降低无效 token 消耗。
- 处理前先去噪、收敛上下文，处理时保留关键证据。
- 代码修改前必须完成调用链完整性确认。

统一触发规则（节省 token 与默认入口合并）：
- 先判断是否为代码工程语境（任一成立即进入该 skill）：
  - 用户意图/上下文明确为编程类：`编写/修改/调试/排查/重构/迁移/解释代码`
  - 或出现“多文件排查/跨模块分析/查 bug/先定位后改”类任务形态
  - 或用户明确要求“减少 token”“总结噪音”“去除噪音”
  - 或用户明确要求输出“简洁、可追溯、可复盘”
 - 任一成立后，若未显式要求“暂不压缩”：
  - 直接切到本模式，不额外停顿确认，开始任务收敛。
- 非触发边界：非工程/非编程咨询（如纯生活对话、一般知识问答、纯情绪化闲聊）不自动切入，除非用户显式点名该模式处理。

冲突优先级子条款（模糊命令决策）：
- 若出现“查下这段/看下这行”等短促表达：只要伴随路径、片段、错误、调用链、文件名或模型代码上下文，即触发；否则先回传一个简短澄清问题再决定。
- 不允许“无输入触发”：仅凭“查下/看下”这类空泛表达，不判定为高确定性编程任务，不默认触发。
- 若用户明确下达“用/不用 codex-noise-filter”指令，显式意图优先级高于所有模糊匹配规则。

优先规则（强制）：
1. 先读取并遵循全局规则文件：
   - `/Users/lilinhan/.codex/AGENTS.md`
2. 若存在项目级 AGENTS，按以下优先级追加：
   - 项目根 `AGENTS.md`（如存在）> 全局规则
3. 无法读取 AGENTS 时，先暂停高风险改代码动作，先澄清约束。

执行模板：
1. 任务封装（必须先发）
- 生成任务胶囊（3~6行）：
  - 目标一句话
  - 允许改动范围
  - 禁止改动项
  - 预期产物
  - 预计工具读文件预算（行数上限和文件上限）

2. 目标压缩与路径收敛
- 先做元信息定位，不读源码正文：
  - 通过搜索/索引找目标符号、关键词、文件候选
  - 只保留 top-N（默认 20）候选
- 只读最小片段（建议 200~300 行窗口）以确认上下文
- 对重复/冗余输出进行摘要过滤，默认不输出完整日志原文

3. 调用链强制确认（仅在涉及代码修改前）
- 若涉及“改代码/重构/重命名/移动/删除”必须完成调用链确认后再执行：
  3.1 定位主修改点（定义点）
    - 文件、类/方法、行为边界、输入输出
  3.2 追踪直接调用方（Direct Callers）
    - 同步给出每个调用方的文件和作用
  3.3 追踪间接链路（Indirect/Route）
    - controller/service/dao/脚本/任务调度/配置入口
  3.4 影响面评估
    - API/DTO/权限/事务/缓存/配置/数据库字段
  3.5 回滚点确认
    - 修改最小补丁 + 回滚文件清单
- 未完整确认调用链，不得进入修改步骤；先回报缺口并继续追链。

4. 改动执行（最小单元）
- 每次只改一个最小闭环：
  - 一个文件改动 -> 一个行为目标
- 任何无关文件不得修改
- 每次改动前后给出“受影响证据（行号级）+ 不变项确认”

5. 结果输出（抗噪）
- 输出顺序固定为：
  - 结论（1段）
  - 已确认调用链（完整路径）
  - 执行动作（按步骤）
  - 关键证据（最少 3 条）
  - 风险与边界
  - 下一步验证建议
- 禁止把工具原始大段日志放入答复正文，优先给摘要与命中片段。

6. 上下文预算控制
- 单次文件读取默认上限：
  - 非必要：200~300 行
  - 关键段：最多 500 行
- 重复读取同一文件时，复用区间缓存记录（已读区间不重复输出）
- 任务接近上下文阈值时自动触发“上下文压缩点”并给用户确认是否继续
- 上下文压缩时不能丢弃任务和记忆核心：调用链状态、已确认约束、待改文件、关键证据与回滚方案必须完整保留。

6.1 上下文压缩保底流程（必做）
- 先生成并输出 Context Capsule（上下文胶囊），包括以下 4 类：
  - 任务边界：目标一句话、当前阶段、禁止项
  - 约束与状态：AGENTS 关键规则摘要、调用链是否闭环、未闭环项
  - 证据锚点：文件路径 + 行号区间 + 事实结论（不放过程日志）
  - 记忆锚点：已改动项、回滚清单、下一步动作
- 输出格式固定：不超过 12 行，且在新会话可直接接续。
- 继续操作前，必须要求用户确认并贴出 Capsule（如：`按 Capsule 继续`）。
- 若必须切新会话，第一条必须完整重放 Capsule；不允许用“延续上文”替代。
- 回退保底：若未能完整重建上述 4 类，任务必须以“待补全”结束，不得直接进入修改。

6.2 人工引导介入处理（防遗忘）
- 用户在执行中插入新指令时，默认不重置任务链，只执行“人类引导增量”：
  1) 先输出 `当前状态快照`（≤6行）：
     - 任务目标与已完成子任务
     - 当前执行阶段（定位/验证/修改/复核）
     - 已确认的调用链状态（已闭环/未闭环）
     - 当前未完成清单
  2) 将用户新指令映射为“本次追加目标”，不覆盖原任务；未明确取消原任务时，原任务保持挂起状态。
  3) 在处理引导任务前，先给出与原任务的关系判断：
     - `继续主任务`、`并行执行`、`暂时转向`、`替代执行`。
  4) 输出 1-2 条冲突检测：若与既定调用链或约束冲突则要求澄清。
  5) 执行新指令后，必须补充 `Context Capsule` 的更新项，保证主任务进度可追踪。
- 严禁在人工引导后做“无端重置”，包括：
  - 重复检索同一调用链再做一次从零推导
  - 忘记已确认调用链与回滚点
  - 丢失已读文件锚点与证据索引

7. 失败与回退策略
- 任何高风险或回归风险动作必须报告：
  - 当前证据链是否完整
  - 失败场景与回退步骤
  - 是否符合 AGENTS 限制
- 出现调用链不完整、规则冲突、边界不确定时，先降级为只读诊断，不直接修改。

常用语模板（可直接使用）：
- `先做调用链确认，再决定改动`（调用链模式）
- `当前读取范围已收敛到 X 个文件，预估新增 token ~Y`（预算模式）
- `改动前调用链未闭环，先补充`（阻断模式）

禁止行为：
- 不得为了省 token 而跳过：
  - AGENTS 读取
  - 调用链确认
  - 影响范围说明
- 不得只给“结论判断”而不提供依据来源。

交付标准：
- 任务完成质量：链路完整、行为一致、边界清晰、验证可复现。
- token 节省标准：避免重复全文读取，删去噪音型重复输出，保留最小证据链。
